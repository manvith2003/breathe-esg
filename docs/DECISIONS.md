# DECISIONS.md — Every Ambiguity Resolved

## SAP Integration Format

**Decision: Pipe-delimited flat file (|) from a custom ABAP extract, not native IDoc.**

**What I researched:** SAP can expose procurement/fuel data via IDoc (fixed-width hierarchical segments E1EDK01/E1EDP01), OData services (/sap/opu/odata/), BAPIs (BAPI_PO_GETDETAIL), or custom ABAP Z-programs writing to flat files.

**What I chose:** The pipe-delimited flat file from a Z-program.

**Why:** Native IDoc fixed-width files require an EDI middleware layer (SAP PI/PO, Dell Boomi, MuleSoft) to parse — these are separate licensed products. OData and BAPI require SAP Basis-level connectivity (RFC, VPN, service user). In the real world, when a sustainability team asks "can you get me the fuel data?", the answer 99% of the time is a pipe or CSV file sitting in an SFTP folder, generated by an ABAP developer's Z-program querying MSEG (material movements) and EKKO/EKPO (purchasing documents). This is the shape the data actually arrives in.

**What I'd ask the PM:** Which plants are in scope? Is there an existing ABAP extract already running, or do we need to commission one? Are there plant codes not in my lookup table?

**What I handled:** German column aliases (Menge, Einheit, Werk, Buchungsdatum), DD.MM.YYYY date format, mixed units (L, GAL, M3, KWH, KG), plant code lookup, automatic fuel type classification.

**What I ignored:** Multi-company-code scenarios, purchasing info records (vendor-material pricing), batch management (CHARG), cost object hierarchies.

## Utility Electricity Format

**Decision: Green Button–style portal CSV with billing period columns.**

**What I researched:** Utility data arrives via: (1) portal CSV export (most common for UK clients — EDF Energy, British Gas, E.ON all offer this), (2) Green Button Connect My Data API (US standard, uncommon in UK), (3) PDF bills (standard for SMEs, unusable without OCR), (4) utility API (rare — requires a bespoke integration per provider).

**What I chose:** Portal CSV export.

**Why:** PDF OCR (option 3) has >10% error rate on poorly formatted utility bills — unacceptable for audit data. The Green Button API (option 2) is primarily US-focused. Portal CSV is what a facilities manager actually emails to the sustainability team. It captures the key challenge: non-calendar billing periods and unit variance (kWh vs MWh).

**What I handled:** Non-calendar billing periods (29-day cycles crossing month boundaries), MWh → kWh conversion, estimated reading flag, tariff codes, MPAN/meter ID, flexible column aliases.

**What I'd ask the PM:** Which utility providers? Is the data by meter point or by site? Do they want market-based (renewable energy certificates) or location-based Scope 2 calculation?

**What I ignored:** Market-based Scope 2 (requires renewable energy certificate data), Scope 3 Cat 8 (upstream T&D losses), time-of-use pricing for carbon intensity variation.

## Corporate Travel Format

**Decision: Concur Expense File Export CSV (bulk export).**

**What I researched:** Concur exposes travel data via: (1) Expense Reports API v4 (OAuth2, JSON), (2) Expense File Export (configurable CSV, no API key needed), (3) direct database extract via SAP Analytics (enterprise only). Navan (formerly TripActions) has a similar bulk export format.

**What I chose:** The Expense File Export CSV.

**Why:** The OAuth2 API requires a registered Concur app, scopes provisioned by the tenant admin, and tenant-specific OAuth credentials — weeks of setup for each client. The Expense File Export is what a travel manager actually downloads from the Reports module and sends to sustainability. It has the same data. The interesting engineering problem is the same either way: distance is often missing for flights (you only get IATA codes), and spend data needs to be converted to activity data.

**What I handled:**
- Flights: explicit distance → haversine IATA fallback → error if neither
- Hotels: explicit nights → check_in/check_out derivation → error if neither
- Ground: explicit distance → analyst flag if missing
- Seat class: economy/business → different DEFRA EF (business flights emit ~2x economy)
- Radiative forcing: included in DEFRA EFs (already baked into the factor, per DEFRA guidance)

**What I'd ask the PM:** Do expense reports capture seat class? Are employees booking through a managed travel platform or personal cards? Is there a TMC (Travel Management Company) we can get segment-level data from?

**What I ignored:** Hotel location-based factors (country-specific EFs), rail travel, commuting (Scope 3 Cat 7), supply chain flights, freight.

## Emission Factor Source

**Decision: UK DEFRA Greenhouse Gas Reporting — Conversion Factors 2023.**

**Why DEFRA over EPA:** The client is described as an enterprise client in the UK context (SAP, utility portals, Concur). DEFRA is the UK government's official source and is also widely used internationally for business travel. The EPA GHG Emission Factors Hub is the equivalent for US-based reporting. ICAO Carbon Calculator is aviation-specific but DEFRA's aviation factors already incorporate ICAO methodology with radiative forcing adjustment.

**What would change in production:** Factors should be parameterized per reporting year (2023, 2024, etc.) and jurisdiction. A real system would have a factor library with `valid_from/valid_to` dates and the ability to recalculate historical records when factors are updated.

## Scope Classification

**Decision: Follow GHG Protocol Corporate Standard strictly.**

- SAP fuel combustion (diesel, petrol, gas, LPG) → **Scope 1** (direct)
- Utility electricity → **Scope 2** (location-based, UK grid factor)
- Business travel (flights, hotels, ground) → **Scope 3, Category 6**
- Unrecognized SAP materials → **Scope 3, Category 1** (purchased goods), auto-flagged

**What I'd ask the PM:** Does the client report Scope 2 on a location-based or market-based basis? (Requires renewable energy certificate data for market-based.) Do they have any Scope 3 Category 15 (investments) reporting requirements?

## Audit Lock Semantics

**Decision: LOCKED records are fully immutable at the API layer.**

Any PATCH, approve, reject, or flag request on a LOCKED record returns HTTP 400 with a clear error message. The only path to modification is deleting the record and re-ingesting — which itself creates a new audit trail entry.

**Why:** Once submitted to auditors, records cannot be retroactively changed. This is the standard in financial audit practice. The django-simple-history trail means even if a record is deleted, its history is preserved.

## Date Handling

**SAP:** Posting date (Buchungsdatum) = the GL posting date, which is the correct carbon accounting date (when the fuel was consumed/procured).

**Utility:** Billing period start = `activity_date`. The full period is stored in `reporting_period_start/end`. This is because electricity consumption happens throughout the billing period, not on a single day.

**Travel:** Transaction date = when the expense was incurred. For flights, this is the departure date. For hotels, this is check-in date.

## Multi-Currency

**Decision: Not normalized. Amounts stored as-is from source.**

Concur travel data arrives in the transaction currency (USD, EUR, GBP, etc.). The emission calculation doesn't require currency conversion because we use activity-based factors (km, nights) not spend-based factors. Currency is stored for reference only.

**What I'd ask the PM:** Do they need spend-based Scope 3 calculations (some categories like insurance, IT services require spend-based EFs)? If yes, we'd need FX rates by date.

---

## Outlier Detection

**Decision: Flag rows where quantity > 3σ from batch mean.**

Implemented in `ingestion/services.py`. Only runs when a batch has ≥ 4 rows (statistically meaningless with fewer). These rows get `status=FLAGGED, flag_reason=outlier_value` and are highlighted in the review UI.

**Why 3σ:** Conservative threshold that catches data entry errors (e.g., 18,500 liters vs typical 1,500) without over-flagging normal variation.
